# Button Guidelines

## 目的
UTab の操作ボタンを「何を押すべきか」が一目で分かる状態に統一する。

## バリアント運用ルール
- `primary`: その領域で最も重要な実行操作。原則1つまで。
- `secondary`: 同等重要だが主操作ではない実行操作。
- `ghost`: 補助操作（戻る、閉じる、詳細表示、展開、コピーなど）。
- `danger`: 破壊的操作（削除、全削除）。

## 画面ごとの運用
- フォーム系カード: `保存/更新/作成` を `primary`、`キャンセル` は `ghost`。
- リスト行操作: `編集` は `ghost`、`削除` は `danger`。
- タブ切り替え・表示切り替え: ボタンではなく「セグメントUI」として扱う。

## セグメントUIの指針
- 選択中: 塗りつぶし強調より「薄い強調背景 + ブランド色文字」を優先。
- 非選択: 境界線あり、低コントラスト文字。
- 同一行に複数セグメントを置く場合、角丸・高さ・余白を揃える。

## 避けるべき状態
- 1ブロック内に `primary` が複数並ぶ。
- ページ遷移タブと実行ボタンが同じ見た目で混在する。
- `danger` を削除以外に使う。

## Reloadの原則
- ページ主Reloadは1つ（ページヘッダー右上）を原則とする。
- セクションReloadは「独立データ境界」「重い再計算」「権限差異」がある場合のみ許容する。
- Reloadは更新対象を `aria-label` で明示する（例: `大会一覧を再読み込み`）。
- Reload近傍に最終更新時刻を表示し、更新対象が最新化されたかを可視化する。
- 保存/作成/削除の成功後は対象セクションを自動再取得し、手動Reload依存を下げる。

## 今回の反映
- 管理画面の `入力方式` は entity 間で共通状態化。
- 参加者画面の `対戦表/スピーカー/ジャッジ` と `カード/テーブル` は同系統セグメントUIに調整。
- 個別大会の管理画面（`/admin/:tournamentId` 配下）も同一philosophyで、タブ/アクティブ表示/ボタン優先度を統一。
